# Project Setup: Markdown Parser (учебный мини-проект)

Этот файл предназначен для ИИ-агента,
который должен произвести **только инфраструктурный сетап проекта**. Вся логика парсера
пишется автором проекта самостоятельно, вручную, без помощи ИИ в написании кода парсера.

---

## 1. Цель проекта

Учебный мини-проект: самостоятельная реализация парсера упрощённого Markdown на TypeScript,
без использования сторонних библиотек для самого парсинга (никаких `marked`, `remark`,
`markdown-it` и т.п.). Проект нужен исключительно для практики и роста как разработчика —
не для продакшена.

Пользовательский сценарий: пользователь загружает `.md` файл через `<input type="file">` на
фронтенде → файл отправляется на сервер (Next.js API route) → сервер читает содержимое файла
и передаёт его в парсер → парсер строит AST (абстрактное синтаксическое дерево) → AST
возвращается на клиент в виде JSON → React-компонент рекурсивно обходит AST и рендерит его
в JSX (без использования `dangerouslySetInnerHTML`).

## 4. Структура проекта

```
/lib
  /parser
    tokenizer.ts        # Токенизатор: raw text -> список токенов (плоский массив)
    block-parser.ts      # Блочный парсер: список токенов -> AST (дерево, без инлайн-разметки)
    inline-parser.ts     # Инлайн-парсер: raw-текст узла -> список инлайн-узлов (bold/italic/link/code)
    pipeline.ts          # Сборка: raw text -> tokenizer -> block-parser -> inline-parser -> готовое AST
    types.ts             # Все TypeScript-типы: Token (discriminated union), AST-узлы (discriminated union)
  /parser/__tests__
    tokenizer.test.ts
    block-parser.test.ts
    inline-parser.test.ts
    pipeline.test.ts

/app
  /api/parse/route.ts    # API route: принимает файл (FormData), читает в строку, вызывает pipeline, возвращает AST как JSON
  /page.tsx               # Главная страница: input[type=file], отправка на /api/parse, получение AST
  /components/MarkdownRenderer.tsx  # Рекурсивный обход AST -> JSX (без dangerouslySetInnerHTML)
```

Все файлы внутри `/lib/parser/*.ts` (кроме `types.ts`) агент создаёт **пустыми заготовками**
с сигнатурами функций-заглушек и комментарием `// TODO: implement`, БЕЗ какой-либо реализации
логики — эта часть будет писаться автором вручную, шаг за шагом.

Файл `types.ts` тоже оставить как заготовку с комментарием, что типы будут определены
автором самостоятельно (discriminated union для Token и AST-узлов) — агент не должен
предугадывать и сам придумывать финальную структуру типов.

## 5. Согласованный Scope (что парсер должен поддерживать в итоге)

**В scope (v1):**

- Заголовки `#` — `######`
- Параграфы (обычный текст)
- Bold `**text**`, italic `*text*`
- Инлайн-код `` `code` ``
- Блок кода (` ``` `, многострочный, без подсветки синтаксиса)
- Списки: unordered (`-`/`*`) и ordered (`1.`), **без вложенности**
- Ссылки `[text](url)`

**Вне scope (сознательно, v1):**

- Таблицы, изображения, вложенные списки, зачёркивание, HTML внутри Markdown, escaping
  спецсимволов, горизонтальные линии.

Спецификация: собственная упрощённая грамматика. CommonMark используется только как
справочник для сверки интуиции в неочевидных случаях, полное соответствие CommonMark —
не цель.

## 6. Согласованная архитектура пайплайна

```
Raw text (string)
  → Tokenizer            (построчная классификация, discriminated union Token[])
  → Block Parser          (строит дерево контейнеров: Document → Heading/Paragraph/List/CodeBlock/...)
  → Inline Parser          (применяется точечно к текстовым узлам: Paragraph/Heading/ListItem;
                             CodeBlock содержимое НЕ проходит инлайн-парсинг)
  → готовое AST (JSON-сериализуемое дерево)
  → передача на клиент (через API route)
  → React-рендерер         (рекурсивный обход AST -> JSX, без dangerouslySetInnerHTML)
```

Ключевые архитектурные решения, зафиксированные заранее:

- Токенизатор не строит вложенность — только плоский список токенов.
- Блочный парсер использует стек либо рекурсивный спуск (выбор — на усмотрение автора,
  делается осознанно при реализации).
- Инлайн-разбор — отдельный проход, не смешивается с блочным парсингом.
- Для представления Token и AST-узлов используется TypeScript discriminated union
  (поле `type` как дискриминант), а не единый interface с опциональными полями.
- Приём файла на сервере — через `FormData` и `request.formData()` / `file.text()`
  (стандартные встроенные API, не библиотеки).

## 7. Чего агент НЕ должен делать

- Не писать реализацию токенизатора, блочного парсера, инлайн-парсера, рендерера или
  pipeline — только сигнатуры-заглушки с TODO.
- Не устанавливать npm-пакеты для Markdown-парсинга.
- Не придумывать финальную структуру типов в `types.ts` — только оставить файл с пометкой,
  что типы будут определены автором.
- Не усложнять сетап лишними инструментами (state management, UI-kit, CSS-in-JS и т.п.) —
  проект должен оставаться максимально простым инфраструктурно.

## 8. Definition of Done для сетапа

- `npm run dev` поднимает Next.js проект без ошибок, дефолтная страница открывается.
- Структура папок и заготовочные файлы созданы согласно разделу 4.
- Тестовый раннер (Vitest/Jest) настроен и `npm run test` (или аналог) успешно запускается
  (даже если тесты пока пустые/заглушечные).
- `tsconfig.json` настроен в строгом режиме.
- Никакие сторонние библиотеки для парсинга не установлены.
